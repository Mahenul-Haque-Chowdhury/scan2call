### 2.2 Models

```prisma
// ─────────────────────────────────────────────
// USERS
// ─────────────────────────────────────────────

model User {
  id                  String              @id @default(cuid())
  email               String              @unique
  emailVerified       Boolean             @default(false)
  emailVerifiedAt     DateTime?
  phone               String?             @unique
  phoneVerified       Boolean             @default(false)
  phoneVerifiedAt     DateTime?
  passwordHash        String
  firstName           String
  lastName            String
  role                Role                @default(CUSTOMER)
  avatarUrl           String?
  timezone            String              @default("Australia/Sydney")
  
  // Notification preferences
  notifyOnScan        Boolean             @default(true)
  notifyViaSms        Boolean             @default(true)
  notifyViaEmail      Boolean             @default(true)
  notifyViaPush       Boolean             @default(false)
  
  // Stripe link
  stripeCustomerId    String?             @unique
  
  // Account state
  isSuspended         Boolean             @default(false)
  suspendedAt         DateTime?
  suspendedReason     String?
  lastLoginAt         DateTime?
  
  // Soft delete
  deletedAt           DateTime?
  
  // Timestamps
  createdAt           DateTime            @default(now())
  updatedAt           DateTime            @updatedAt
  
  // Relations
  tags                Tag[]
  scans               Scan[]              // Scans of the user's tags (as owner)
  orders              Order[]
  subscription        Subscription?
  payments            Payment[]
  communicationLogs   CommunicationLog[]  // As tag owner
  refreshTokens       RefreshToken[]
  adminAuditLogs      AdminAuditLog[]     // Actions taken BY this admin

  @@index([email])
  @@index([stripeCustomerId])
  @@index([deletedAt])
  @@map("users")
}

model RefreshToken {
  id          String    @id @default(cuid())
```

### 2.3 Schema Design Notes

**Token generation**: The `Tag.token` field is a 12-character base62 string (a-z, A-Z, 0-9) generated via `crypto.randomBytes(16)` and encoded. This yields 62^12 (approximately 3.2 x 10^21) possible values -- non-sequential, non-guessable, and URL-friendly. The public scan URL becomes `https://scan2call.com.au/scan/{token}`.

**Privacy by design**: The `Scan` model stores a hashed IP (SHA-256 with a server-side salt), never the raw IP. Latitude/longitude are rounded to city-level precision (2 decimal places, approximately 1.1km accuracy). No finder PII is ever persisted. The `CommunicationLog` stores Twilio session references, not real phone numbers.

**Soft deletes**: Applied to `User`, `Tag`, and `Product` via the `deletedAt` nullable timestamp pattern. Prisma middleware or a query extension will auto-filter soft-deleted records on read queries.

**Tag states**: The lifecycle is `INACTIVE` (generated, not yet sold/activated) then `ACTIVE` (linked to an owner, normal operation) then `LOST` (owner toggles lost mode, finder sees contact options) then `FOUND` (optional intermediate state when a finder reports found) and back to `ACTIVE` once the owner confirms. `DEACTIVATED` is a terminal state for stolen/broken/decommissioned tags.

## 4. Key Design Decisions

### 4.1 Anonymous Communication Relay -- Technical Architecture

The relay uses **Twilio Proxy** as its foundation, supplemented by **Twilio Voice** for browser-based calling.

**Flow for phone-based relay (SMS and voice):**

1. Finder scans QR code, arrives at `https://scan2call.com.au/scan/{token}`.
2. Frontend calls `GET /api/v1/scan/{token}` to load tag info (type, label, contact options). No owner PII is returned.
3. Finder chooses "Call" or "Text" and enters their phone number (or uses browser calling).
4. Frontend calls `POST /api/v1/communication/call` (or `/sms`) with `{ token, finderPhone }`.
5. Backend:
   a. Looks up the tag by token, verifies it is in ACTIVE or LOST status.
   b. Retrieves the owner's real phone from the `User` record (server-side only, never sent to client).
   c. Creates a **Twilio Proxy Session** with a short TTL (default: 10 minutes for calls, 30 minutes for SMS).
   d. Adds two **Participants** to the session: the owner's real phone and the finder's real phone. Twilio assigns each a masked proxy number.
   e. Twilio initiates the call/SMS through the proxy. Both parties see only the Twilio proxy number.
   f. The backend stores the session SID in `CommunicationLog` but does NOT store the finder's real phone number.
6. When the session TTL expires, the proxy number association is dissolved. Neither party can contact the other through that number again.

**Flow for browser-based calling (no phone number needed):**

1. Finder taps "Call from Browser" on the scan page.
2. Frontend calls `POST /api/v1/communication/browser-call` with `{ token }`.
3. Backend generates a **Twilio Client Access Token** (short-lived, scoped) and returns it.
4. Frontend initializes the Twilio Client JS SDK, places a call.
5. The call hits a **TwiML App** webhook on the backend which dials the owner's phone through Twilio, displaying the Twilio proxy number as the caller ID.
6. The owner sees a call from the Twilio number, never from the finder's identity (the finder has no phone number involved at all).

**WhatsApp relay (premium tier):**

Uses the Twilio WhatsApp Business API. A Twilio Proxy session is created similarly, but the channel is set to WhatsApp. The finder sends a WhatsApp message to the Twilio WhatsApp number, which relays it to the owner. This requires the owner to have opted in to WhatsApp contact on that tag.

### 4.2 QR Token Generation and Scan Flow

**Token generation:**

```
crypto.randomBytes(9)  ->  12-character base62 string
```

Nine random bytes = 72 bits of entropy, encoded to base62 (A-Z, a-z, 0-9) yields a 12-character token. This gives approximately 3.2 x 10^21 possible tokens -- infeasible to brute-force, yet short enough for a clean URL. Tokens are generated server-side and checked for uniqueness against the database before insertion (with a retry on collision, though collisions are astronomically unlikely).

**Batch generation** (admin): The `POST /admin/tags/generate` endpoint generates N tokens in a transaction, creates Tag records in `INACTIVE` status, links them to a `TagBatch`, and optionally returns a CSV or a zip of QR code PNGs. QR codes encode the full URL: `https://scan2call.com.au/scan/{token}`.

**QR code encoding:** Each QR code encodes the full scan URL. The QR image is generated server-side using a library such as `qrcode` (npm), rendered as SVG or PNG, with the Scan2Call logo optionally embedded in the center (using error correction level H which tolerates 30% obscuring). Generated images are stored in the S3 bucket under `qr-codes/{batchId}/{token}.png`.

**Full scan flow:**

```
Physical QR tag
  -> Phone camera scans QR
  -> Opens https://scan2call.com.au/scan/{token}
  -> Next.js SSR/ISR page
  -> Calls GET /api/v1/scan/{token}
  -> Returns: { tagType, label, description, ownerFirstName (first name only),
                contactOptions: { call: true, sms: true, whatsapp: false },
                lostMode: true, lostMessage: "Please help me find my dog!" }
  -> Finder sees branded page with contact buttons
  -> Finder taps "Call" or "SMS"
  -> Twilio Proxy session created
  -> Anonymous communication happens
  -> Owner is simultaneously notified (push/SMS/email) that their tag was scanned
```

| Family Members (shared accounts) | No | No | No | Up to 5 |
| Tag Photo | No | Yes | Yes | Yes |

All prices in AUD. Annual billing available at a 20% discount. Free tier requires email verification. Stripe Checkout is used for initial subscription creation. Plan changes (up/down) use the Stripe API with proration. The `Subscription` model stores denormalized limits (`maxTags`, `maxScansPerMonth`, `allowsWhatsApp`, `allowsLocationTracking`) which are synced from Stripe webhook events. This avoids needing to query Stripe for every authorization check.

**Stripe webhook events handled:**

- `checkout.session.completed` -- Provision subscription after initial checkout.
- `invoice.paid` -- Renew subscription period, reset monthly counters.
- `invoice.payment_failed` -- Mark subscription as `PAST_DUE`, notify owner, trigger dunning.
- `customer.subscription.updated` -- Sync plan changes, period dates.
- `customer.subscription.deleted` -- Mark subscription `CANCELLED`, downgrade to free limits.
- `charge.refunded` -- Update `Payment` record with refund details.

### 4.4 File/Image Storage Strategy

**Storage provider:** S3-compatible object storage (AWS S3 or a provider like Cloudflare R2 which has no egress fees -- important for an Australian startup optimising costs).

**Upload flow:** The client never uploads directly to the API server. Instead:

1. Client requests a pre-signed upload URL: `POST /api/v1/media/upload-url` with `{ filename, contentType, context: "tag-photo" }`.
2. Backend validates the request (file type whitelist: JPEG, PNG, WebP; max size: 5MB), generates a pre-signed S3 PUT URL with a 15-minute expiry, and returns it along with the final public CDN URL.
3. Client uploads directly to S3 via the pre-signed URL.
4. Client then submits the CDN URL when updating the tag or profile.

**Bucket structure:**

```
scan2call-media/
├── tag-photos/{userId}/{tagId}.webp
├── avatars/{userId}.webp
├── qr-codes/{batchId}/{token}.png
└── product-images/{productId}/{index}.webp
```

**Image processing:** On upload, a background job (triggered by an S3 event or an API-side queue) resizes and converts images to WebP at multiple sizes (thumbnail 150px, medium 400px, large 800px). The CDN URL pattern supports size variants: `cdn.scan2call.com.au/tag-photos/{userId}/{tagId}_400.webp`.

### 4.5 Rate Limiting and Abuse Prevention

Rate limiting uses a layered approach:

**Layer 1 -- Global (NestJS ThrottlerModule with Redis store):**
- Default: 100 requests per minute per IP.
- Applied via a global guard, overridable per-route with `@Throttle()`.

**Layer 2 -- Endpoint-specific limits:**
- `POST /auth/login`: 5 attempts per 15 minutes per IP (brute force protection).
- `POST /auth/forgot-password`: 3 requests per hour per email.
- `POST /communication/call`: 3 calls per 10 minutes per IP (prevents relay abuse).
- `POST /communication/sms`: 5 SMS per 10 minutes per IP.
- `POST /communication/browser-call`: 3 calls per 10 minutes per IP.
- `GET /scan/:token`: 30 requests per minute per IP (prevents token enumeration).
- `POST /admin/tags/generate`: 5 requests per hour (prevent runaway batch generation).

**Layer 3 -- Business logic abuse detection:**
- If a single tag receives more than 20 scans in an hour from different IPs, flag for review and optionally temporarily disable relay for that tag.
- If a single IP scans more than 10 different tokens in 5 minutes, block that IP for 1 hour (token enumeration attempt).
- CAPTCHA (hCaptcha or Turnstile) on the finder-facing contact form before initiating relay.

**Layer 4 -- Infrastructure:**
- Cloudflare (or equivalent) WAF in front of the application for DDoS protection, bot management, and geographic filtering.
- The webhook endpoints (`/webhooks/stripe`, `/webhooks/twilio/*`) are exempt from rate limiting but protected by signature verification (see below).

### 4.6 Webhook Signature Verification

**Stripe:**

Every incoming request to `POST /webhooks/stripe` is verified using the `stripe.webhooks.constructEvent()` method. This requires:
1. The raw request body (not parsed). NestJS must be configured to preserve raw body for this route (via `rawBody: true` in the NestFactory options or a dedicated middleware).
2. The `stripe-signature` header from the request.
3. The webhook signing secret from the Stripe dashboard (stored in environment variable `STRIPE_WEBHOOK_SECRET`).

If verification fails, the endpoint returns `400 Bad Request` immediately. The webhook controller should be stateless and idempotent -- it uses the Stripe event ID to deduplicate (checking against a `processed_webhook_events` table or a Redis set with TTL).

**Twilio:**

Every incoming request to `/webhooks/twilio/*` is verified using the `twilio.validateRequest()` method (or the `@twilio/webhook` middleware). This verifies:
1. The `X-Twilio-Signature` header.
2. The full URL of the webhook endpoint (must match exactly, including protocol and port).
3. The request body parameters.
4. The Twilio Auth Token (stored in environment variable `TWILIO_AUTH_TOKEN`).

A NestJS guard (`TwilioWebhookGuard`) is applied to all Twilio webhook routes. This guard extracts the signature and validates it before the request reaches the controller. Requests with invalid signatures are rejected with `403 Forbidden`.

### 4.7 Additional Architecture Decisions

**JWT structure:**
- Access token: 15-minute expiry, contains `{ sub: userId, role, email }`. Signed with RS256 (asymmetric) so the frontend can decode without the secret.
- Refresh token: 7-day expiry, opaque (stored in database), rotated on each use (old token revoked, new one issued). Sent as an HttpOnly, Secure, SameSite=Strict cookie.

**CORS:**
- Frontend origin only: `https://scan2call.com.au` and `https://www.scan2call.com.au`.
- The scan page (`/scan/:token`) is same-origin, so no CORS issue.
- Webhook endpoints do not need CORS (server-to-server).

**Australian data residency:**
- Database hosted in `ap-southeast-2` (Sydney) region.
- S3 bucket in the same region.
- CDN with Australian edge nodes.
- This satisfies the Australian Privacy Act requirements for an Australian business handling PII.

**Caching strategy:**
- Product listings: CDN cached (5-minute TTL) via Next.js ISR.
- Scan pages: ISR with on-demand revalidation when tag data changes (the API calls the Next.js revalidation webhook).
- API responses for product/plan listings: Redis cache with 5-minute TTL.
- User-specific data (tags, orders): No caching, always fresh from the database.

**Testing strategy:**
- Backend: Unit tests for services (mocked Prisma, Twilio, Stripe), integration tests for controllers (using `@nestjs/testing` with a test database), E2E tests for critical flows (scan, relay, checkout).
- Frontend: Component tests with React Testing Library, E2E with Playwright for the scan flow and checkout flow.
- Shared: Pure function tests for validation schemas.
